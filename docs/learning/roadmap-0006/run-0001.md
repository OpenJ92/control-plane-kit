# Roadmap 0006 Run 0001 Learning

Date: 2026-07-15
Roadmap: Control Plane Read Interfaces
Branch context: first Roadmap 0006 implementation branch

## Summary

The first Roadmap 0006 run successfully produced a coherent read-interface
surface:

```text
Postgres-backed stores
  -> projection read models
    -> InstanceReadService
      -> FastAPI read routes
      -> CLI read commands
      -> read-only MCP-shaped adapter
```

The architecture held up, but the process learned something important: broad
interface work needs explicit hardening passes. The initial implementation
tests were not enough for the amount of surface area added.

The later hardening pass found real bugs, so hardening must be part of the
loop.

## What Went Well

- The source-of-truth boundary stayed clear.
  - Stores own durable truth.
  - Read services compose store truth.
  - Projections expose read models.
  - FastAPI, CLI, and MCP are adapters.
- No write routes, execution routes, or live control-route calls were added.
- Secret values stayed hidden.
- Address values were redacted by default.
- The MCP adapter was kept transport-neutral instead of pretending we had a
  hosted MCP server and auth story already.
- The final hardening pass improved the implementation without changing the
  architectural shape.

## What Went Wrong

The first implementation leaned too much on happy-path tests.

The original tests proved that the read surfaces worked when:

- workspaces existed,
- graph pointers were populated or intentionally absent,
- limits were positive,
- adapters received normal arguments,
- and projections had fully wired graph examples.

They did not sufficiently prove:

- invalid limits fail consistently;
- missing workspaces fail consistently;
- adapters over the same service preserve the same laws;
- CLI opt-in address exposure actually works and remains opt-in;
- dangling requirement sockets are visible to the operator;
- MCP tools fail loudly for unassigned graph reads;
- service-level invariants are strong enough to protect all adapters.

## Bugs Found During Hardening

### CLI Limit Coercion

The CLI used:

```python
limit=limit or 50
```

That meant:

```text
--limit 0
```

was silently interpreted as the default limit instead of failing. This is a
classic Python truthiness trap. The fix was:

```python
limit=50 if limit is None else limit
```

### Unknown Workspace Reads

Activity and observed-state reads could return empty payloads for missing
workspaces. That violated the source-of-truth law because workspace existence
belongs to `WorkspaceStore`.

The fix was to validate workspace existence at the read-service boundary:

```python
_positive_limit(limit)
self._workspace_store.get(workspace_id)
```

This lets HTTP, CLI, and MCP inherit the same rule.

## Process Lesson

The loop should be:

```text
child issue implementation PR
  -> review pass
    -> hardening pass PR
      -> final roadmap integration validation
```

Hardening is not optional ceremony when a PR or roadmap adds an interface,
persistence, security, runtime, or adapter surface. It is how we discover the
bugs that live between modules.

## How To Rerun Roadmap 0006

When retrying Roadmap 0006 from `develop`, use this structure:

1. Build each child issue as before.
2. After each child PR, review it against:
   - AGENTS module ownership law;
   - ADR 0008 transaction policy;
   - security/redaction;
   - adapter consistency;
   - test depth;
   - documentation truthfulness.
3. Add a hardening PR for the child issue when the surface is broad.
4. At the end of the roadmap, run a whole-roadmap hardening pass anyway.

Do not wait for the user to notice thin tests.

## What To Test Earlier Next Time

For read interfaces:

- missing workspace;
- unassigned current graph;
- unassigned desired graph;
- invalid limits;
- boundary limit values;
- address redaction by default;
- address opt-in only where explicitly supported;
- secret redaction;
- dangling requirement sockets;
- unknown MCP tool names;
- mutation-like MCP tool names;
- adapter consistency over the same service method;
- route auth when token configured;
- route behavior when token unconfigured;
- no writes during reads.

## Design Result

The Roadmap 0006 architecture still looks right.

The important law is:

```text
Interfaces expose the model; they do not define the model.
```

The hardening pass strengthened that law by moving consistency checks into
`InstanceReadService` instead of duplicating them in each adapter.

## Handoff To Future Roadmaps

Future roadmap runs should read this document before implementation.

In particular:

- Roadmap 0007 should not duplicate read projection logic inside planning.
- Planning and approval should use Roadmap 0006 descriptors as the operator
  review lens.
- Any new adapter should get both happy-path and edge-path tests.
- Any broad interface roadmap should expect at least one hardening issue/PR.
