# Roadmap 0006 Run 0002 Learning

Date: 2026-07-15
Roadmap: Control Plane Read Interfaces
Branch context: second execution from `develop` with the updated loop

## Summary

The second Roadmap 0006 run rebuilt the read-interface vertical with the
hardening lesson from run 0001 applied from the start.

The final shape is:

```text
Postgres-backed stores
  -> projection helpers
    -> InstanceReadService
      -> FastAPI read routes
      -> CLI read commands
      -> MCP-shaped read adapter
```

This run reinforced the central law:

```text
Interfaces expose the model; they do not define the model.
```

FastAPI, CLI, and MCP-shaped reads now share the same service boundary. That is
the important design result. Adapters should stay boring.

## What Changed Compared With Run 0001

The updated loop produced smaller child PRs and forced explicit review between
implementation slices.

The hardening work was no longer an afterthought. It arrived as an explicit
issue when the operator graph/read-service surface became broad enough to
deserve a separate edge-case pass.

That prevented two earlier failure modes:

- adapter behavior drifting away from service behavior;
- and happy-path tests pretending the read model was stronger than it was.

## Design Lessons

### One Read Service

The service is the object. The adapters are morphisms out of it.

```text
InstanceReadService
  -> HTTP JSON
  -> CLI JSON
  -> MCP-shaped JSON
```

This should remain true in future roadmaps. Planning, approval, execution, MCP,
and UI code should not duplicate graph projection logic.

### Redaction Belongs Below Adapters

Redaction must be owned by the read model layer. If FastAPI, CLI, and MCP each
redact independently, one of them will eventually leak something.

The current tests assert that graph/operator/control-surface payloads do not
surface actual address-like values.

### Missing Workspace Is Not Empty Workspace

Activity and observed-state reads validate workspace existence. This preserves
source-of-truth boundaries and keeps operator UIs from rendering a missing
workspace as a quiet empty state.

### MCP Is A Vocabulary First

The MCP adapter is intentionally not a network server yet.

That decision keeps Roadmap 0006 within read-interface scope. A future MCP
runtime can wrap `ReadOnlyMcpAdapter` once the hub/instance auth story is ready.

## Testing Lessons

The useful edge tests for this roadmap were:

- missing workspace,
- unassigned graph pointer,
- unknown graph pointer,
- invalid limit,
- boundary limit,
- stale observed state,
- dangling requirement socket,
- unknown route-set warning,
- route auth enabled,
- route auth disabled,
- unknown MCP tool,
- mutation-like MCP tool name,
- CLI query parameters,
- and no secret/address leakage.

Future broad adapter work should start with this kind of list, not end with it.

## Data Engineering Notes

The read services use Postgres-backed stores in tests. That matters because
Roadmap 0006 is not a toy in-memory projection. The package is already obeying
the Postgres unit-of-work direction from ADR 0008 for durable boundaries.

Read routes do not create write transactions, but they still respect ownership:

```text
repository/store truth
  -> service projection
    -> adapter serialization
```

Do not let a future adapter query Postgres tables directly.

## Security Notes

The read surfaces are intentionally conservative:

- no write routes,
- no execution routes,
- no MCP mutation tools,
- bearer or `X-Control-Plane-Token` protection for FastAPI reads when
  configured,
- bounded activity timeline reads,
- and redacted graph/control descriptors.

The next security boundary is not in Roadmap 0006. It is the hub/instance
session model and runtime control-route auth story.

## Handoff To Roadmap 0007

Roadmap 0007 should treat Roadmap 0006 descriptors as the operator review
surface for plans and approvals.

In particular:

- an activity plan should be explainable using operator graph node/socket IDs;
- approval screens should use control-surface descriptors instead of raw
  capability internals;
- plan previews should be able to link to current/desired graph reads;
- and any planning read route should call `InstanceReadService`, not rebuild
  graph projection.

If planning needs a read payload that does not exist, add it to the read service
deliberately rather than creating a planning-only projection.

## Handoff To Roadmap 0010

This handoff was originally numbered Roadmap 0009. After inserting the
control-plane-instance block and recursive-navigation vertical, UI/MCP work is
Roadmap 0010.

Roadmap 0010 can turn the MCP-shaped adapter into a real MCP server, but should
not weaken the current boundary.

The adapter already gives the tool vocabulary. The missing pieces are:

- runtime server hosting,
- auth/session integration,
- transport-level error mapping,
- and operator-facing documentation for AI clients.

Mutation tools should remain absent until planning, approval, and execution
semantics are ready.
