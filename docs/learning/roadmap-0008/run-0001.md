## Existing Truth To Reuse

```text

  -> typed GraphDiff
    -> deterministic ActivityPlan DAG
      -> durable plan
        -> approval request and decision
```

The package also already has useful scaffolding:

- `ActivityRunRecord`, `ActivityEventRecord`, and `ObservationRecord`;
- Postgres tables and append/read methods for runs, events, and observations;
- `DockerRuntimeInterpreter` plus a Docker client boundary;

These are starting points, not finished execution contracts.

## Main Gaps

### Execution Truth Is Still String-Shaped

Run status, event type, observation status, and metadata remain open strings.
There is no execution-request record, claim owner, retry lineage, guarded
transition grammar, or typed failure evidence. Roadmap 0008 must migrate these
records deliberately rather than add parallel models.

### Postgres Stores Are Recorders, Not Execution Repositories

The current methods can append a run or event and list them. They do not claim
queued work, serialize event ordinals, enforce one active run, compare-and-set
status, or distinguish idempotent replay from conflicting intent. Those laws
belong in Postgres primitives and UnitOfWork-backed application services before
an executor exists.

### Current Docker API Is Too Broad

`DockerRuntimeInterpreter.up()` starts an entire runtime and currently reports
started nodes as healthy. `down()` operates at runtime scope. Activity execution
needs narrow node/runtime effects, inspection, explicit health probes, stable
ownership identity, retained-resource policy, and host-publication policy.
Existing recipes and client seams should be preserved while the effect surface
is refactored.

### Control Routes Exist But Need A Safe Client

Package servers expose capability, health, status, logs, targets, active-target,
drain, and observer routes. The server token is optional for development. The
Roadmap 0008 mutation client must require explicit credentials for mutation,
bound time and payloads, reject unsafe redirects, redact tokens, and verify
capabilities before calling routes. Docker/private networking is not auth.

### Saga Callbacks Are Not Durable Syntax

The Pottery Factory saga package has a clean immutable sequential/parallel
program and reverse-completion compensation interpreter. Its activities embed
Python compute and compensate callbacks. That is acceptable for an in-process
saga but cannot be persisted, reconstructed, inspected, or resumed as the
control plane's durable activity language.

The adapted shape should separate syntax from interpretation:

```text
SagaProgram[TypedEffect]
  -> EffectInterpreter[TypedEffect, EffectResult]
    -> external call
```

Callbacks may live inside an interpreter registry, not inside durable program
descriptors. Compensation remains an attempted external action, not rollback.

### ActivityPlan Order Is Not A Scheduler

`ActivityPlan` validates dependency composition and stores a deterministic
topological tuple. Execution still needs ready sets, fan-out, blocked dependents,
step outcomes, restart reconstruction, and a durable cursor. The scheduler must
remain pure and domain-neutral.

## Data-Engineering Boundary

The mandatory rhythm is:

```text
short Postgres transaction: record intent/step start
  -> commit
    -> external effect with bounded timeout
      -> short Postgres transaction: record result/event/observation
```

No transaction may remain open while Docker, HTTP, health, filesystem, cloud,
or secret-provider work executes. A crash after an effect but before result
persistence creates an explicit uncertain state. Exactly-once external effects
are not promised; idempotency keys and reconciliation are required.

## Security Boundary

Execution introduces authenticated mutation, control tokens, Docker access,
network calls, host ports, logs, image trust, retained data, and destructive
cleanup. Every child PR must state new surfaces, authorization, secret handling,
network exposure, destructive behavior, tests, and residual risk.

MCP mutation remains out of scope until the execution command services and CPI
authorization boundary are complete.

## Database Migration Boundary

Roadmap 0007 models only a cutover between pre-provisioned database endpoints.
Roadmap 0008 must require typed readiness evidence before that cutover can run.
It must not infer copy, replication, schema migration, consistency, or safe
retirement from topology. A future migration effect capability may automate
those facts; initially they remain user/provider-managed.

## Stop Conditions

Stop and request a decision if implementation would:

- hold a database transaction over an external effect;
- execute without durable approval/admission;
- rely on optional/empty control authentication for mutation;
- claim exactly-once effects without provider idempotency evidence;
- delete retained data as compensation;
- infer database migration from endpoint cutover;
- duplicate Roadmap 0007 plan or graph models;
- embed opaque callbacks in durable saga descriptors; or
- specialize generic execution around CPI, Auth, Postgres, or one example.

## Mandatory Milestone Gates

Roadmap 0008 must pause for an explicit integration review at the following
points. A gate is not merely a progress update. It is a proof that the laws
needed by the next effectful layer exist in source and tests.

Roadmap 0008 must not be executed as one uninterrupted issue chain. At every
gate, stop implementation, run the gate's full validation, update this learning
document with the source truth and review findings, and report the result before
proceeding. Later issues being planned or ready does not permit crossing a gate.

### Gate A: Durable Execution Truth (`#211` through `#215`)

At this gate the package may admit, claim, and transition an execution attempt,
but it may not perform an external effect. Review closed execution types,
descriptor strictness, schema migration, idempotency, approval freshness,
one-winner claims, event ordinal serialization, retry lineage, and lifecycle
transition laws.

### Gate B: Pure Execution Language (`#216` through `#218`)

At this gate the package has immutable saga syntax, deterministic schedule
derivation, and typed effect capabilities. Review that these modules contain no
Postgres, Docker, HTTP-client, CPI, or application-domain assumptions and that
restart reconstruction is a pure function of the plan and durable evidence.

### Gate C: Durable Fake-Effect Execution (`#219`)

This is the central architecture gate. Prove the complete rhythm with injected
fake effects:

```text
transaction: record step intent
  -> commit
    -> external effect
      -> transaction: record result and advance run
```

Inject crashes before the effect, after the effect, and before result commit.
The last case must become explicit uncertainty rather than blind replay. No
real Docker or HTTP interpreter may be connected before this gate passes.

### Gate D: Real Runtime Effects (`#220` through `#222`)

At this gate authenticated bounded HTTP control, narrow Docker lifecycle
effects, and truthful health observations exist. Review secret handling, SSRF
and redirects, payload and timeout bounds, Docker ownership and retention,
private versus host endpoints, and the distinction between process state,
transport reachability, and application health.

### Gate E: Failure Semantics And Acceptance Corpus (`#223` and `#224`)

At this gate pause, resume, terminal failure, uncertainty, and compensation are
durable and visible across the reusable scenario corpus. Review reverse
completion order, compensation failure, retry admission, blocked dependents,
database readiness gates, and event partial-order assertions.

### Gate F: Live Proof And Closeout (`#225` through `#227`)

At this gate one approved graph transition executes through the real pipeline,
changes a Docker-backed router, records observations, and cleans up without
damaging retained state. Review live evidence, prior recipe compatibility,
security/data findings, public exports, and the exact Roadmap 0009 handoff.

Every gate report must state:

- the capability that now exists;
- the objects, transformations, and executable laws introduced;
- the tests and live evidence protecting those laws;
- provisional behavior and remaining risk;
- security and data-engineering findings;
- any deviation from this dry run; and

- Prove identical claims converge, competing workers do not both win, ordinals

### Capability Reached

Gate A establishes durable execution truth without executing runtime effects:

```text
approved non-empty ActivityPlan
  -> guarded ExecutionRequest admission
    -> concurrency-safe worker claim
      -> admitted ActivityRun attempt
        -> typed lifecycle transition
          -> canonical ActivityEvent + operation evidence
```

The package can now admit, claim, open, retry, start, pause, resume, complete,
fail, compensate, and cancel an execution attempt. Every operator command is
one explicit Postgres transaction. No command in this layer accepts an effect
interpreter or calls Docker, HTTP, the filesystem, a cloud provider, or a
health endpoint.

An empty plan has a deliberately narrower meaning:

```text
empty GraphDiff
  -> ActivityPlan(())
    -> durable planning conclusion: no runtime work is required
      -/-> ExecutionRequest
```

An empty plan remains valid, inspectable planning truth. Admission rejects it
because creating a request, worker claim, run, or event would falsely claim
that executable work exists.

### Objects And Transformations

Gate A replaced open execution strings and recorder-style persistence with a
closed execution vocabulary and guarded application services. The central
lifecycle interpretation is:

```python
RunLifecycleCommand
  -> decide_run_transition(command, current_status)
  -> RunTransition(expected, replacement, event_kind)
  -> PostgresUnitOfWork
  -> RunLifecycleResult(request, run, event, action)
```

The durable objects now include:

- closed request, run, event, observation, failure, claim, and retry values;
- strict versioned descriptor codecs;
- execution requests with approval and intent identity;
- worker claims with leases;
- admitted run attempts with explicit retry lineage;
- canonical, ordinal `ActivityEvent` journal entries; and
- existing `OperationAction` rows as actor-attributed idempotency receipts.

`RUN_OPENED` was added to the closed event vocabulary. Reusing
`REQUEST_CLAIMED` for a retry would have made the event journal describe a
request operation that did not occur. Forward migration rebuilds the closed
Postgres check constraint to admit this constructor while continuing to reject
unknown event values.

### Executable Laws

The Docker/Postgres suite now proves:

- unknown descriptor versions, variants, lifecycle strings, and secret-shaped
  evidence fail closed;
- existing Roadmap 0007 rows migrate without fabricated admission facts;
- request claims have exactly one concurrent winner;
- event and operation ordinals serialize across real Postgres connections;
- compare-and-set run transitions reject stale state;
- one command commits request/run state, event, and operation evidence together;
- a late evidence-write failure rolls back every earlier write;
- current authorization is evaluated before idempotent replay;
- reuse of an idempotency key with different intent fails explicitly;
- retry creates a new attempt linked to the latest failed or partially failed
  run and cannot branch from an older attempt;
- compensation replaces the failed attempt's terminal timestamp with truthful
  active and final compensation timing;
- cancellation clears claim ownership and remains exactly replayable; and
- every lifecycle command accepts exactly its declared source statuses.

Integrated Gate A validation ran `git diff --check` and `./test.sh`; all 368
tests passed against Docker and real Postgres.

### Architecture Review

The implementation preserves the intended separation:

```text
planning truth -> approval -> admission -> run lifecycle
                                      |
                                      `-- no effects yet
```

The lifecycle transition table is pure data interpreted by a pure function.
The application service owns transaction entry, commit, and rollback. Stores
lock and mutate rows but never commit. No CPI, Auth, Router, Docker, Postgres
role, or scenario name specializes the execution algebra.

No competing request, run, event, planner, or receipt model was introduced.
The existing `ActivityEvent` relation remains the future saga journal, and the
existing operation history remains the user-command evidence ledger.

### Security Review

- execution admission revalidates current approval and policy-derived scope;
- lifecycle commands require a current `execution:operate` authority;
- claimed request ownership is checked before mutation;
- authorization is checked before idempotent replay, so an old receipt does not
  become a capability token;
- bounded evidence rejects secret-shaped fields; and
- Gate A introduces no network, Docker socket, token transport, redirect, SSRF,
  image, host-port, or retained-resource surface.

The worker authority remains an application-boundary value rather than a full
identity-provider integration. Roadmap 0008 must continue treating it as
verified input supplied by the future authenticated CPI command boundary, not
as a credential parser.

### Data-Engineering Review

Every command follows the Roadmap law:

```text
one operator command = one explicit Postgres transaction
```

All participating stores share the UnitOfWork connection. Request and run rows
are locked deliberately, database constraints provide final concurrency
authority, and application services perform the only commit. There is still no
transaction-over-effect path because no effect exists at this gate.

Schema installation now also acts as a forward migration for the closed event
constraint. The migration test begins with the older constraint, proves the
new constructor is accepted after installation, and proves an invented value
is still rejected.

### Test-Integrity Review

No test was skipped, softened, or changed merely to accept incorrect behavior.
The execution-store fixture formerly used an empty plan for convenience; it now
uses a minimal typed `StartRuntime` activity because empty plans are correctly
non-executable. The dedicated admission test retains the stronger assertion:
an empty plan is persistable planning truth, admission raises a conflict, and
no execution request is written.

The lifecycle tests use real Postgres for concurrency, row locking, rollback,
migration, idempotency, and read-projection visibility. Pure tests exhaust the
closed transition table. Mocks did not replace application behavior.

### Provisional Behavior And Residual Risk

- Claim leases are durable evidence, but Gate A does not renew leases or run a
  recovery worker. Later scheduler/coordinator work must make expired and
  uncertain ownership explicit.
- A replay returns the original command event and action together with the
  current durable run record. Read surfaces must continue distinguishing
  original command evidence from current aggregate state.
- Compensation lifecycle states now exist, but no compensation program or
  external compensating effect exists until Gates B through E.
- `FailureEvidence` records bounded operator-facing facts; provider response
  bodies and secrets must remain outside it.
- Admission proves current policy and readiness references, not the continued
  truth of an external system. Gate C and Gate D must record operational
  observations around effects.
- Database endpoint cutover remains a plan over pre-provisioned endpoints. Gate
  A provides no migration, replication, consistency, or retirement guarantee.

### Roadmap Deviations

The Gate A capability matches the dry run. Focused hardening added one event
constructor (`RUN_OPENED`) and reused the operation-action ledger for lifecycle
idempotency rather than creating another receipt table. Both changes reduce
semantic duplication. No external effect or Roadmap 0009 packaging work was
pulled forward.

### Gate B Readiness

Gate B is unblocked from Gate A's perspective. It may consume admitted run and
event types while defining pure saga syntax, scheduler reconstruction, and
effect capabilities. It must not import Postgres stores, lifecycle command
services, Docker, HTTP, CPI, or application-domain assumptions into that pure
language.
