## Main Gaps

### Execution Truth Is Still String-Shaped

Run status, event type, observation status, and metadata remain open strings.
There is no execution-request record, claim owner, retry lineage, guarded
transition grammar, or typed failure evidence. Roadmap 0008 must migrate these
records deliberately rather than add parallel models.

### Postgres Stores Are Recorders, Not Execution Repositories

The current methods can append a run or event and list them. They do not claim
queued work, serialize event ordinals, enforce one active run, compare-and-set
status, or distinguish idempotent replay from conflicting intent. Those laws
belong in Postgres primitives and UnitOfWork-backed application services before
an executor exists.

### Current Docker API Is Too Broad

`DockerRuntimeInterpreter.up()` starts an entire runtime and currently reports
started nodes as healthy. `down()` operates at runtime scope. Activity execution
needs narrow node/runtime effects, inspection, explicit health probes, stable
ownership identity, retained-resource policy, and host-publication policy.
Existing recipes and client seams should be preserved while the effect surface
is refactored.

### Control Routes Exist But Need A Safe Client

Package servers expose capability, health, status, logs, targets, active-target,
drain, and observer routes. The server token is optional for development. The
Roadmap 0008 mutation client must require explicit credentials for mutation,
bound time and payloads, reject unsafe redirects, redact tokens, and verify
capabilities before calling routes. Docker/private networking is not auth.

### Saga Callbacks Are Not Durable Syntax

The Pottery Factory saga package has a clean immutable sequential/parallel
program and reverse-completion compensation interpreter. Its activities embed
Python compute and compensate callbacks. That is acceptable for an in-process
saga but cannot be persisted, reconstructed, inspected, or resumed as the
control plane's durable activity language.

The adapted shape should separate syntax from interpretation:

```text
SagaProgram[TypedEffect]
  -> EffectInterpreter[TypedEffect, EffectResult]
    -> external call
```

Callbacks may live inside an interpreter registry, not inside durable program
descriptors. Compensation remains an attempted external action, not rollback.

### ActivityPlan Order Is Not A Scheduler

`ActivityPlan` validates dependency composition and stores a deterministic
topological tuple. Execution still needs ready sets, fan-out, blocked dependents,
step outcomes, restart reconstruction, and a durable cursor. The scheduler must
remain pure and domain-neutral.

## Data-Engineering Boundary

The mandatory rhythm is:

```text
short Postgres transaction: record intent/step start
  -> commit
    -> external effect with bounded timeout
      -> short Postgres transaction: record result/event/observation
```

No transaction may remain open while Docker, HTTP, health, filesystem, cloud,
or secret-provider work executes. A crash after an effect but before result
persistence creates an explicit uncertain state. Exactly-once external effects
are not promised; idempotency keys and reconciliation are required.

## Security Boundary

Execution introduces authenticated mutation, control tokens, Docker access,
network calls, host ports, logs, image trust, retained data, and destructive
cleanup. Every child PR must state new surfaces, authorization, secret handling,
network exposure, destructive behavior, tests, and residual risk.

MCP mutation remains out of scope until the execution command services and CPI
authorization boundary are complete.

## Database Migration Boundary

Roadmap 0007 models only a cutover between pre-provisioned database endpoints.
Roadmap 0008 must require typed readiness evidence before that cutover can run.
It must not infer copy, replication, schema migration, consistency, or safe
retirement from topology. A future migration effect capability may automate
those facts; initially they remain user/provider-managed.

## Stop Conditions

Stop and request a decision if implementation would:

- hold a database transaction over an external effect;
- execute without durable approval/admission;
- rely on optional/empty control authentication for mutation;
- claim exactly-once effects without provider idempotency evidence;
- delete retained data as compensation;
- infer database migration from endpoint cutover;
- duplicate Roadmap 0007 plan or graph models;
- embed opaque callbacks in durable saga descriptors; or
- specialize generic execution around CPI, Auth, Postgres, or one example.

### Capability Reached

The package can now take one durably admitted and claimed `ActivityRun`,
reconstruct its pure schedule from canonical history, and make bounded progress
through an injected fake effect interpreter:

```text
ActivityPlan x ActivityEvent journal
  -> SagaJournalProjection
  -> ExecutionSchedule
  -> PreparedEffect
  -> transaction(STEP_STARTED intent)
  -> commit
  -> fake EffectInterpreter
  -> transaction(STEP_SUCCEEDED | STEP_FAILED + observations)
  -> durable run settlement
```

No Docker, HTTP, filesystem, cloud, or other real runtime effect is connected.
The coordinator cannot advance `current_graph_id`. A separate guarded command
may do so only after complete durable success evidence proves the exact base and
desired graph identities.

### Objects And Transformations

The central Gate C objects are:

```python
@dataclass(frozen=True)
class SagaJournalProjection:
    state: SagaState
    in_flight: tuple[ActivityEventRecord, ...] = ()
    uncertain: tuple[ActivityEventRecord, ...] = ()


@dataclass(frozen=True)
class PreparedEffect:
    request: EffectRequest
    interpreter: EffectInterpreter


class ExecutionCoordinator:
    def execute(
        self,
        command: ExecuteActivityRun,
    ) -> ExecutionCoordinatorResult: ...
```

`project_activity_journal()` is a pure adapter from the one canonical durable
event history into the Roadmap 0008 saga state. It does not persist another
journal. `derive_schedule()` remains the dependency interpreter. `prepare_effect()`
freezes capability support before intent is committed, while
`dispatch_prepared_effect()` is called only after that transaction closes.

### Executable Laws

The Docker/Postgres suite now proves:

```text
intent transaction closed before effect starts

effect result
  -> result event + observations in one new transaction

crash after intent, before effect
  -> active lease
  -> IN_FLIGHT
  -> eventual explicit uncertainty
  -> no blind effect replay

crash after effect, before durable result
  -> eventual explicit uncertainty
  -> no blind effect replay

durable success/failure result + missing run settlement
  -> journal-driven settlement
  -> no effect replay

unsupported capability
  -> STEP_UNSUPPORTED
  -> interpreter is never invoked

active leased intent + concurrent coordinator
  -> IN_FLIGHT
  -> exactly one effect invocation
  -> no manufactured uncertainty

complete durable run success + exact graph identities
  -> guarded compare-and-set current graph advancement

partial, failed, uncertain, unsupported, stale, or foreign evidence
  -> no current graph advancement
```

### Review Findings And Corrections

The implementation and hardening reviews found and corrected:

- capability preflight could otherwise become a time-of-check/time-of-use
  split; `PreparedEffect` now freezes the support decision;
- successful or failed result persistence could survive a process crash before
  run settlement; replay now settles from journal evidence without dispatch;
- result evidence now identifies the exact intent event and effect idempotency
  key;
- an uncertain saga step is unresolved but not actively in flight; the journal
  projection now represents those predicates independently;
- bounded multi-step progress returns a typed `PROGRESSED` result without
  settling the run or inventing an activity identity;
- sequential crash tests did not prove live concurrent observation, so #276
  added a blocking fake interpreter and two real coordinator invocations; and
- raw current-graph pointer mutation was too broadly available, so #273 added
  AST-enforced call ownership around the guarded workflow.

No test was skipped, weakened, converted to a snapshot, or replaced with a
mock of application behavior. The fake interpreter is an explicit
implementation of the effect protocol. Its transaction tracker raises if an
effect begins while any `PostgresUnitOfWork` is active.

### Data Engineering Review

The coordinator obeys the external-effect law:

```text
short transaction: revalidate locked run/request/plan and append intent
  -> commit and close
    -> bounded effect
      -> short transaction: revalidate open intent and append result/evidence
```

The request and run rows are locked only inside short command transactions.
Event ordinals remain database-serialized. The concurrent hardening test uses
separate Postgres connections and proves the observer sees committed intent.
Observations and the successful result event commit atomically. Current graph
advancement atomically records its operation evidence and compare-and-set
pointer change in a separate operator command.

### Security Review

Gate C adds no network listener, HTTP client, Docker socket access, host port,
or secret resolver. Effect requests carry references and bounded descriptors,
not secret values. Worker identity and `execution:operate` scope are checked
before dispatch, and claimed-request ownership is revalidated in each command
transaction.

Real control-route authentication, SSRF policy, redirects, response bounds,
Docker ownership labels, retained-resource handling, and host publication
remain Gate D work. The fake interpreter does not provide evidence for those
properties.

### Provisional Behavior And Residual Risk

- Attempt leases are a conservative crash detector, not proof that an external
  provider stopped. Expired missing-result attempts become operator-visible
  uncertainty and are not retried automatically.
- Provider-level idempotency is represented by a stable key but cannot be
  proven until concrete interpreters honor it.
- Gate C executes one ready activity at a time. The pure scheduler supports
  fan-out ready sets, but parallel effect dispatch is deliberately deferred.
- Compensation, pause/resume policy, retry admission, and operator resolution
  of uncertainty remain Gate E work.
- Database endpoint cutover remains routing to a pre-provisioned endpoint. It
  does not perform migration, copying, replication, or consistency checks.
- `ActivityEventRecord` remains the durable saga journal. Any later need for
  richer provider evidence must extend that canonical vocabulary rather than
  create a competing execution store.

### Validation And Readiness

Merged Gate C validation runs in Docker against Postgres. The final hardening
suite contains 470 passing tests. `git diff --check`, architecture dependency
policies, call-ownership policies, and test-integrity policies also pass.

Gate D is unblocked only to implement #245, #220, #221, and #222 through the
existing typed effect boundary. It must not replace the coordinator, journal,
scheduler, current-graph command, or transaction rhythm proven here.

### `#213`: Claims, Ordinals, And Transition Concurrency

**Existing source to migrate**

- Run/event stores have no `FOR UPDATE`, claim primitive, compare-and-set
  transition, or ordinal allocator.
- The Roadmap 0007 action ordinal implementation is the nearest tested pattern,
  but execution claims have stronger worker-concurrency semantics.

**Expected implementation**

- Add one-winner queue claim, guarded run transition, serialized next event
  ordinal, lease expiry evidence, and explicit retry-attempt creation.
- Prefer Postgres row locking plus constraints over process locks.
- Keep policy out of stores: primitives enforce concurrency; command services
  decide whether a transition is authorized and meaningful.

**Tests and difficulty**

- Use two independent Postgres connections and real concurrent transactions.
- Prove identical claims converge, competing workers do not both win, ordinals
  remain unique, stale compare-and-set fails, and rollback releases claims.
- Do not test this with mocks; concurrency is a database property.

**Handoff:** `#214` and `#215` may rely on these primitives inside one caller-
owned UnitOfWork.

## #329 Decision Log: Recovery Concurrency And Expired Claims

Recovery command intent now includes the exact journal version observed by the
operator:

```python

    run_id: str
    expected_worker_id: str
    expected_event_ordinal: int
    recovery: RecoveryDecisionRecord

```

The workflow checks exact idempotent replay first. A replay therefore returns
the original result even though its recorded consequence advanced the journal.

mutations. Renewal preserves the worker, takeover records a replacement, and

- uncertain forward or compensation effects never replay blindly;
- stable idempotency identity survives coordinator reconstruction;
- no UnitOfWork or Postgres lock spans an adapter call;
- no-op planning creates no approval, admission, run, or effect;
- review-blocked and externally readiness-gated plans do not execute;
- graph truth advances only after exact complete success evidence, including an
  explicit succeeded uncertainty resolution;
- a failed uncertainty resolution cannot advance graph truth;
- scenario code cannot import stores, SQL, scheduler internals, private
  coordinator operations, mocks, or parallel read models;
- only explicit reviewed optional-dependency skips exist.

### Review Findings And Corrections

Gate E intentionally stopped whenever validation exposed a semantic question.
The material corrections were:

1. Run terminal timestamps were replaced by append-only attempt records so
   compensation no longer erases the time at which forward execution ended.
2. Expired claims gained explicit renewal, takeover, and abandonment commands
   instead of silent clearing or reclaim.
3. Compensation admission became a pure saga event. A coordinator flag could
   not reconstruct scheduling after restart.
4. Recovery commands gained journal-version preconditions and exact
   idempotency, preventing a stale operator decision from winning against new
   evidence.
5. The current unreleased schema was tightened in place. Speculative legacy
   compatibility values and destructive schema-install DDL were removed.
6. Bounded durable evidence kept its JSON-shaped language. Internal tuples are
   converted explicitly to arrays only at the descriptor boundary.
7. Operator-confirmed uncertain success became valid advancement evidence while
   retaining the original uncertain event. Confirmed failure remains
   disqualifying.
8. Forward versus compensation intent became the closed `EffectPurpose` value;
   the acceptance provider no longer parses idempotency-key text.
9. Scenario architecture became an AST-enforced boundary so acceptance
   composition cannot silently absorb application or persistence authority.

No correction broadened an open string language, removed an assertion, added a
skip, replaced application behavior with a mock, or allowed a transaction to
cross an effect.

Three implementation issues (#317, #318, and #330) were stale-open even though
their reviewed PRs were merged. The milestone review closed them with PR and
validation evidence. All Gate E implementation and hardening children are now
closed; #327 is the milestone closeout itself.

### Gate F Composition Finding

The desired application program can be formed from existing canonical services
without another domain model:

```text
Deploy
  = Plan
  -> suspend for Approve
  -> Admit
  -> Claim
  -> Execute
  -> suspend for Recovery when required
  -> Advance
```

The existing stage mapping is:

| Stage | Canonical service and command | Canonical result |
| --- | --- | --- |
| Plan | operation session + desired-graph command + `ActivityPlanningCommandService(RequestActivityPlan)` + approval request | `GraphTransitionPlanningResult` today; Gate F may replace this example-owned aggregate with a thin application receipt |
| Approve | `ApprovalCommandService(DecidePlanApproval)` | `ApprovalDecisionResult` |
| Admit | `ExecutionAdmissionCommandService(RequestPlanExecution)` | `ExecutionAdmissionResult` |
| Claim | `RunLifecycleCommandService(ClaimAndOpenActivityRun)` followed by `StartActivityRun` | canonical `RunLifecycleResult` values |
| Execute | `ExecutionCoordinator(ExecuteActivityRun)` | `ExecutionCoordinatorResult` |
| Advance | `CurrentGraphAdvancementCommandService(AdvanceCurrentGraph)` | `CurrentGraphAdvancementResult` |

Approval is a required suspension after planning because the approver may be a
different human with different authority. Recovery is a required suspension
whenever execution returns failed, paused, uncertain, uncompensated, or
compensation-required evidence. Neither suspension needs mutable facade state:
the planning/session/approval/run identifiers and complete event history are
durable, so a later process reconstructs the continuation from canonical
stores and projections.

`Advance` is not an unconditional finalizer. It changes current graph truth
only when the admitted request, plan, worker, expected pointer, graph ownership,
terminal run projection, exact per-activity success evidence, and reconstructed
saga completion all agree.

One transition model covers every deployment shape:

```text
initial deploy = Deploy(EmptyGraph, desired)
update         = Deploy(current, desired)
teardown       = Deploy(current, EmptyGraph)
no-op          = Deploy(graph, graph)
```

The no-op returns durable planning evidence and stops before approval. A
review-blocked transition also stops before approval. An externally gated
transition stops before admission until readiness evidence exists.

The example helpers `plan_graph_transition()` and the orchestration portion of
`run_execution_scenario()` are composition plumbing suitable for replacement
by Gate F. Their calls target canonical services. The scenario expectation
evaluator and synthetic effect program remain example/test tools. No helper
contains scheduler, recovery, transaction, or graph-advancement business rules;
the one planning decision it currently makes, whether a nonempty ready plan
requests approval, belongs in the future `Plan` application stage.

Recommended Gate F organization:

```text
control_plane_kit/application/deploy/
  values.py       # transition intent and thin suspension/completion receipts
  stages.py       # Plan, Approve, Admit, Claim, Execute, Advance callables
  program.py      # parameterized Deploy composition
```

Each stage should be a parameterized callable class over the canonical command
services. `Deploy` should compose those callables and return closed outcomes
such as awaiting approval, awaiting readiness, awaiting recovery, completed,
or no changes. Receipts may carry canonical identifiers and results but must
not duplicate plans, runs, events, schedules, failures, or projections.

Gate F architecture policy should enforce that the application deployment
package:

- imports public workflow/read/effect interfaces but not Postgres stores or SQL;
- calls public `execute()` boundaries, never private coordinator methods;
- does not commit, roll back, mutate graph pointers, or inspect scheduler
  internals;
- does not define another plan, run, event, recovery, projection, or effect
  result model;
- does not infer purpose, state, capability, or operation from identifier text;
- advances only through `CurrentGraphAdvancementCommandService`;
- keeps human approval and recovery as explicit resumable outcomes;
- uses the canonical coordinator and real adapters in the live proof.

This organization is a readability layer over proven semantics, not a new
execution engine.

### Gate G #454 immutable configuration-artifact algebra

#454 introduced configuration as a pure graph language rather than Docker
metadata or an arbitrary host mount:

```python
@dataclass(frozen=True, order=True)
class ConfigurationArtifact:
    artifact_id: str
    target_path: str
    media_type: ConfigurationMediaType
    content: str = field(compare=False, repr=False)
    file_mode: ConfigurationFileMode = ConfigurationFileMode.READ_ONLY
    content_digest: str = field(init=False)
```

The value admits only normalized absolute container file paths, bounded UTF-8
content, closed media types, and read-only modes. It rejects reserved runtime
paths, URL credentials, private-key material, secret-shaped assignments, and
secret-shaped JSON keys. The SHA-256 digest is derived from content and verified
when reconstructing a descriptor.

The internal and durable forms remain deliberately distinct:

```text
ConfigurationArtifact
  -> descriptor() containing string leaves
    -> existing graph/plan/effect JSONB boundaries
      -> ConfigurationArtifact.from_descriptor(...)
```

Python tuples and enums therefore remain in the algebra, while stores, API,
MCP, CLI, and future UI boundaries receive deterministic JSON-compatible arrays
and strings. There is no new relational column and no typed Python object is
written directly into Postgres.

Artifacts now propagate through the complete pure pipeline:

```text
DockerImageImplementation
  -> MaterializedNode
    -> DeploymentGraph.Node
      -> GraphDiff(configuration-artifacts)
        -> ActivityPlan(ReconcileNode)
          -> pinned ImplementationMaterial
```

Changing content changes the derived digest and produces explicit graph work.
Duplicate artifact identities or target paths fail at authoring and graph
construction. Exact content and digest are pinned before execution; host paths
are absent from the language. Configuration remains separate from retained
`DataResource` values and from secret material reserved for #458.

The architecture policy now declares `configuration` as a dependency-free
foundation and permits only `implementations`, `topology`, and `effects` to
depend on it. The first complete test run correctly rejected the undeclared new
package root; the policy was extended explicitly rather than weakening its
exhaustiveness assertion.

Review confirmed:

- descriptors are deterministic and reject missing, extra, malformed, or
  digest-mismatched data;
- configuration changes survive graph reconstruction, diff, planning, and
  pinned effect materialization;
- secret-shaped and traversal inputs fail closed;
- no host filesystem path, arbitrary bind mount, secret value, compatibility
  version, second configuration model, skip, or weakened assertion was added;
- no Postgres transaction, runtime effect, retained-data behavior, or cleanup
  behavior changed in this pure issue.

### Gate G #455 owned Docker configuration realization

#455 interprets the exact `ConfigurationArtifact` values pinned by effect
materialization. Each artifact becomes one package-owned ephemeral Docker
volume whose ownership fingerprint includes its node, target path, media type,
digest, and file mode:

```python
DockerConfigurationMount(
    artifact=artifact,
    volume_name=volume_name,
    ownership=_configuration_ownership(request, node, artifact),
)
```

The Docker adapter writes a fixed `content` file through a digest-pinned Python
helper image. Content travels over standard input rather than process arguments.
The helper has no network, a read-only root filesystem, no capabilities, and
`no-new-privileges`; it atomically replaces a temporary file only after flushing
it. The application container receives only a read-only named-volume subpath:

```text
type=volume,source=<owned volume>,target=<container path>,
volume-subpath=content,readonly
```

Before starting the application, the interpreter independently reads and hashes
the materialized file and requires the exact pinned digest. Identical sequential
replay reuses both volume and container. Changed configuration changes the
ownership fingerprint and fails closed rather than overwriting live material.
Helper failure is explicit uncertainty, and the owned partial volume remains
discoverable for canonical node-resource cleanup.

Configuration cleanup remains separate from retained data cleanup. Removing an
ephemeral node preflights the container and every configuration volume before
the first destructive call, removes the container, then removes only its proven-
owned configuration volumes. Named `DataResource` volumes remain untouched.
Late Docker failure produces uncertainty rather than claiming complete cleanup.

Security review added two constraints beyond the first implementation pass:

- container target path segments use a closed Docker-mount-safe grammar, so
  graph data cannot inject comma-separated `--mount` options;
- the helper image is pinned by SHA-256 digest rather than a mutable tag.

Test-integrity review found one old ownership fixture coupled to call position.
The corrected test locates the semantic `run` event and preserves every exact
ownership-label assertion; no assertion, skip, or application behavior was
weakened.
