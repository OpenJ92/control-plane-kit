## Main Gaps

### Execution Truth Is Still String-Shaped

Run status, event type, observation status, and metadata remain open strings.
There is no execution-request record, claim owner, retry lineage, guarded
transition grammar, or typed failure evidence. Roadmap 0008 must migrate these
records deliberately rather than add parallel models.

### Postgres Stores Are Recorders, Not Execution Repositories

The current methods can append a run or event and list them. They do not claim
queued work, serialize event ordinals, enforce one active run, compare-and-set
status, or distinguish idempotent replay from conflicting intent. Those laws
belong in Postgres primitives and UnitOfWork-backed application services before
an executor exists.

### Current Docker API Is Too Broad

`DockerRuntimeInterpreter.up()` starts an entire runtime and currently reports
started nodes as healthy. `down()` operates at runtime scope. Activity execution
needs narrow node/runtime effects, inspection, explicit health probes, stable
ownership identity, retained-resource policy, and host-publication policy.
Existing recipes and client seams should be preserved while the effect surface
is refactored.

### Control Routes Exist But Need A Safe Client

Package servers expose capability, health, status, logs, targets, active-target,
drain, and observer routes. The server token is optional for development. The
Roadmap 0008 mutation client must require explicit credentials for mutation,
bound time and payloads, reject unsafe redirects, redact tokens, and verify
capabilities before calling routes. Docker/private networking is not auth.

### Saga Callbacks Are Not Durable Syntax

The Pottery Factory saga package has a clean immutable sequential/parallel
program and reverse-completion compensation interpreter. Its activities embed
Python compute and compensate callbacks. That is acceptable for an in-process
saga but cannot be persisted, reconstructed, inspected, or resumed as the
control plane's durable activity language.

The adapted shape should separate syntax from interpretation:

```text
SagaProgram[TypedEffect]
  -> EffectInterpreter[TypedEffect, EffectResult]
    -> external call
```

Callbacks may live inside an interpreter registry, not inside durable program
descriptors. Compensation remains an attempted external action, not rollback.

### ActivityPlan Order Is Not A Scheduler

`ActivityPlan` validates dependency composition and stores a deterministic
topological tuple. Execution still needs ready sets, fan-out, blocked dependents,
step outcomes, restart reconstruction, and a durable cursor. The scheduler must
remain pure and domain-neutral.

## Data-Engineering Boundary

The mandatory rhythm is:

```text
short Postgres transaction: record intent/step start
  -> commit
    -> external effect with bounded timeout
      -> short Postgres transaction: record result/event/observation
```

No transaction may remain open while Docker, HTTP, health, filesystem, cloud,
or secret-provider work executes. A crash after an effect but before result
persistence creates an explicit uncertain state. Exactly-once external effects
are not promised; idempotency keys and reconciliation are required.

## Security Boundary

Execution introduces authenticated mutation, control tokens, Docker access,
network calls, host ports, logs, image trust, retained data, and destructive
cleanup. Every child PR must state new surfaces, authorization, secret handling,
network exposure, destructive behavior, tests, and residual risk.

MCP mutation remains out of scope until the execution command services and CPI
authorization boundary are complete.

## Database Migration Boundary

Roadmap 0007 models only a cutover between pre-provisioned database endpoints.
Roadmap 0008 must require typed readiness evidence before that cutover can run.
It must not infer copy, replication, schema migration, consistency, or safe
retirement from topology. A future migration effect capability may automate
those facts; initially they remain user/provider-managed.

## Stop Conditions

Stop and request a decision if implementation would:

- hold a database transaction over an external effect;
- execute without durable approval/admission;
- rely on optional/empty control authentication for mutation;
- claim exactly-once effects without provider idempotency evidence;
- delete retained data as compensation;
- infer database migration from endpoint cutover;
- duplicate Roadmap 0007 plan or graph models;
- embed opaque callbacks in durable saga descriptors; or
- specialize generic execution around CPI, Auth, Postgres, or one example.

### Capability Reached

The package can now take one durably admitted and claimed `ActivityRun`,
reconstruct its pure schedule from canonical history, and make bounded progress
through an injected fake effect interpreter:

```text
ActivityPlan x ActivityEvent journal
  -> SagaJournalProjection
  -> ExecutionSchedule
  -> PreparedEffect
  -> transaction(STEP_STARTED intent)
  -> commit
  -> fake EffectInterpreter
  -> transaction(STEP_SUCCEEDED | STEP_FAILED + observations)
  -> durable run settlement
```

No Docker, HTTP, filesystem, cloud, or other real runtime effect is connected.
The coordinator cannot advance `current_graph_id`. A separate guarded command
may do so only after complete durable success evidence proves the exact base and
desired graph identities.

### Objects And Transformations

The central Gate C objects are:

```python
@dataclass(frozen=True)
class SagaJournalProjection:
    state: SagaState
    in_flight: tuple[ActivityEventRecord, ...] = ()
    uncertain: tuple[ActivityEventRecord, ...] = ()


@dataclass(frozen=True)
class PreparedEffect:
    request: EffectRequest
    interpreter: EffectInterpreter


class ExecutionCoordinator:
    def execute(
        self,
        command: ExecuteActivityRun,
    ) -> ExecutionCoordinatorResult: ...
```

`project_activity_journal()` is a pure adapter from the one canonical durable
event history into the Roadmap 0008 saga state. It does not persist another
journal. `derive_schedule()` remains the dependency interpreter. `prepare_effect()`
freezes capability support before intent is committed, while
`dispatch_prepared_effect()` is called only after that transaction closes.

### Executable Laws

The Docker/Postgres suite now proves:

```text
intent transaction closed before effect starts

effect result
  -> result event + observations in one new transaction

crash after intent, before effect
  -> active lease
  -> IN_FLIGHT
  -> eventual explicit uncertainty
  -> no blind effect replay

crash after effect, before durable result
  -> eventual explicit uncertainty
  -> no blind effect replay

durable success/failure result + missing run settlement
  -> journal-driven settlement
  -> no effect replay

unsupported capability
  -> STEP_UNSUPPORTED
  -> interpreter is never invoked

active leased intent + concurrent coordinator
  -> IN_FLIGHT
  -> exactly one effect invocation
  -> no manufactured uncertainty

complete durable run success + exact graph identities
  -> guarded compare-and-set current graph advancement

partial, failed, uncertain, unsupported, stale, or foreign evidence
  -> no current graph advancement
```

### Review Findings And Corrections

The implementation and hardening reviews found and corrected:

- capability preflight could otherwise become a time-of-check/time-of-use
  split; `PreparedEffect` now freezes the support decision;
- successful or failed result persistence could survive a process crash before
  run settlement; replay now settles from journal evidence without dispatch;
- result evidence now identifies the exact intent event and effect idempotency
  key;
- an uncertain saga step is unresolved but not actively in flight; the journal
  projection now represents those predicates independently;
- bounded multi-step progress returns a typed `PROGRESSED` result without
  settling the run or inventing an activity identity;
- sequential crash tests did not prove live concurrent observation, so #276
  added a blocking fake interpreter and two real coordinator invocations; and
- raw current-graph pointer mutation was too broadly available, so #273 added
  AST-enforced call ownership around the guarded workflow.

No test was skipped, weakened, converted to a snapshot, or replaced with a
mock of application behavior. The fake interpreter is an explicit
implementation of the effect protocol. Its transaction tracker raises if an
effect begins while any `PostgresUnitOfWork` is active.

### Data Engineering Review

The coordinator obeys the external-effect law:

```text
short transaction: revalidate locked run/request/plan and append intent
  -> commit and close
    -> bounded effect
      -> short transaction: revalidate open intent and append result/evidence
```

The request and run rows are locked only inside short command transactions.
Event ordinals remain database-serialized. The concurrent hardening test uses
separate Postgres connections and proves the observer sees committed intent.
Observations and the successful result event commit atomically. Current graph
advancement atomically records its operation evidence and compare-and-set
pointer change in a separate operator command.

### Security Review

Gate C adds no network listener, HTTP client, Docker socket access, host port,
or secret resolver. Effect requests carry references and bounded descriptors,
not secret values. Worker identity and `execution:operate` scope are checked
before dispatch, and claimed-request ownership is revalidated in each command
transaction.

Real control-route authentication, SSRF policy, redirects, response bounds,
Docker ownership labels, retained-resource handling, and host publication
remain Gate D work. The fake interpreter does not provide evidence for those
properties.

### Provisional Behavior And Residual Risk

- Attempt leases are a conservative crash detector, not proof that an external
  provider stopped. Expired missing-result attempts become operator-visible
  uncertainty and are not retried automatically.
- Provider-level idempotency is represented by a stable key but cannot be
  proven until concrete interpreters honor it.
- Gate C executes one ready activity at a time. The pure scheduler supports
  fan-out ready sets, but parallel effect dispatch is deliberately deferred.
- Compensation, pause/resume policy, retry admission, and operator resolution
  of uncertainty remain Gate E work.
- Database endpoint cutover remains routing to a pre-provisioned endpoint. It
  does not perform migration, copying, replication, or consistency checks.
- `ActivityEventRecord` remains the durable saga journal. Any later need for
  richer provider evidence must extend that canonical vocabulary rather than
  create a competing execution store.

### Validation And Readiness

Merged Gate C validation runs in Docker against Postgres. The final hardening
suite contains 470 passing tests. `git diff --check`, architecture dependency
policies, call-ownership policies, and test-integrity policies also pass.

Gate D is unblocked only to implement #245, #220, #221, and #222 through the
existing typed effect boundary. It must not replace the coordinator, journal,
scheduler, current-graph command, or transaction rhythm proven here.

### `#213`: Claims, Ordinals, And Transition Concurrency

**Existing source to migrate**

- Run/event stores have no `FOR UPDATE`, claim primitive, compare-and-set
  transition, or ordinal allocator.
- The Roadmap 0007 action ordinal implementation is the nearest tested pattern,
  but execution claims have stronger worker-concurrency semantics.

**Expected implementation**

- Add one-winner queue claim, guarded run transition, serialized next event
  ordinal, lease expiry evidence, and explicit retry-attempt creation.
- Prefer Postgres row locking plus constraints over process locks.
- Keep policy out of stores: primitives enforce concurrency; command services
  decide whether a transition is authorized and meaningful.

**Tests and difficulty**

- Use two independent Postgres connections and real concurrent transactions.
- Prove identical claims converge, competing workers do not both win, ordinals
  remain unique, stale compare-and-set fails, and rollback releases claims.
- Do not test this with mocks; concurrency is a database property.

**Handoff:** `#214` and `#215` may rely on these primitives inside one caller-
owned UnitOfWork.

### Principal Finding: Real Effects Need Pinned Material

Gate C's `EffectRequest` deliberately carries operation identity rather than
provider mechanics. That is sufficient for a fake interpreter. Real adapters
need immutable runtime material:

```text
ActivityPlanRecord
  x exact desired DeploymentGraph
  x PlannedActivity
  -> MaterializedEffectRequest
```

The compiled graph already contains generic node implementation metadata,
runtime membership, network metadata, endpoints, sockets, edge assignments,
and health declarations. Issue #295 introduces one pure interpreter over that
data. The coordinator must load the plan and its pinned desired graph through
the same short UnitOfWork, materialize before durable intent, then close the
transaction before any provider call.

This prevents three architectural failures:

1. graph drift cannot change the meaning of an admitted plan;
2. Docker, HTTP, and health adapters do not import or query stores; and
3. each provider does not invent a competing topology interpreter.

Secret values remain outside the material. Only opaque secret references may
cross this boundary.

### Contract Mutation Finding

`EnvironmentContract.apply_patch()` currently mutates live values before
derived-resource work completes. `rebuild_derived()` publishes a replacement
before disposing the old resource. Gate D therefore needs:

```text
immutable candidate
  -> prepare every rebuildable resource
  -> publish values + resources + version once
  -> dispose replaced resources
  -> report cleanup uncertainty without rolling back publication
```

Publication is the commit point. Preparation failure preserves the old
projection. Post-publication disposal failure is durable/operator-visible
cleanup evidence, not permission to overwrite history or pretend rollback.

### HTTP Control Finding

The block-control server already separates application data routes from the
deploy protocol, but an empty configured token permits direct control-route
access. Gate D mutation clients and servers must fail closed, resolve opaque
credentials only at the adapter boundary, and explicitly constrain SSRF,
redirects, endpoint scope, timeouts, and response size. Docker-private
networking is not authentication.

### Docker Finding

The legacy Docker client removes same-name containers before start and has no
ownership-label proof. Gate D must inspect first, label owned resources, reject
foreign collisions, model retention explicitly, and keep host publication
opt-in. It must not infer data retention from image names or reuse whole-runtime
`up()` as an activity-level interpreter.

### Observation Finding

Typed observation records, Postgres history, freshness projection, and
coordinator result persistence already exist. Gate D should extend this path,
not create a second observation workflow. Process state, transport
reachability, application health, readiness, stale evidence, and unknown state
must remain distinct. Observations never rewrite desired graph truth.

### #317 Closed Recovery Decisions And Authorization

`FailureCategory` already classifies effect outcomes, while
`ExecutionWorkerAuthority` authorizes ordinary execution. Neither is sufficient
to choose a recovery path. Add a closed `RecoveryDecision` language and a
distinct recovery authority value. Legal decisions must be interpreted from
run state and journal evidence, while authorization remains a separate check.

The canonical lifecycle service must be extended rather than replaced.
Retryability is evidence, not permission, and ordinary worker scope must not
authorize uncertainty resolution or acceptance of uncompensated loss.

### #318 Pinned Typed Compensation Programs

`PlannedActivity` is persisted compiler output and is already the durable value
corresponding to a saga leaf. Add compensation specification there rather than
creating a separate compensation plan store.

The compensation specification must distinguish:

```text
CompensationSpec
  = Compensate(operation, material_source)
  | NoCompensationRequired
  | NonCompensatable(reason)

material_source = BaseGraph | DesiredGraph
```

The pinned source is necessary. Stopping a newly started node normally
materializes from the desired graph, while restoring a changed socket normally
materializes from the base graph. Current graph truth must never choose that
source after admission.

`initial_state_for_plan()` currently marks every saga step as having no
compensation. It must derive compensation availability from the persisted plan
specification. Structural parallel nodes remain scheduler structure and do not
acquire aggregate compensation effects.

### #319 And #328 Durable Recovery History

`ActivityEvent` remains the one append-only execution journal. Existing event
names currently overload run-level and step-level compensation. Gate E must
introduce distinct closed events and preserve both original and compensation
failure evidence.

`ActivityRun` remains a reconstructable current projection. There is no
persisted resume cursor. Recovery decision attribution belongs in typed event
evidence, including operator identity, bounded reason, and authority reference.

Issue #328 was added because this event-language change requires its own
Postgres proof: supported old schemas must migrate idempotently without opening
constraints, deleting rows, or confusing run and step events. Unknown legacy
strings must continue to fail closed.

### #320 And #329 Transactional Recovery Commands

The existing `RunLifecycleCommandService` and execution stores already own
guarded transitions. They must be deliberately extended or migrated; Gate E
must not introduce another lifecycle service with competing transition rules.

Every recovery decision is one operator command and therefore one explicit
Postgres UnitOfWork. Decision event, operation action, and run projection move
atomically. External effects remain outside that transaction.

Issue #329 was added to isolate the concurrency proof. Retry, compensation,
uncompensated acceptance, and pause must have exactly one winner across real
database connections. Expired claims are never silently cleared: takeover,
renewal, or abandonment is an explicit authorized event-producing command.

### #321 And #330 Durable Compensation Coordination

The existing coordinator already implements the required boundary:

```text
short transaction: durable intent
  -> bounded effect outside Postgres
    -> short transaction: durable result
```

Extend it to interpret admitted compensation work in reverse durable completion
order. The coordinator chooses ready work but does not choose recovery policy.
It reconstructs state from `ActivityEvent` history after restart and does not
persist an incidental cursor.

Issue #330 was added because compensation has the same dangerous crash windows
as forward effects. A crash after effect completion but before result storage
must become visible uncertainty, not blind replay. Earlier successful
compensations and the original forward failure remain visible if a later
compensation fails.

Read models remain projections of canonical journal evidence. They contain no
independent mutable truth and expose no secret references, private headers, or
adapter material.
