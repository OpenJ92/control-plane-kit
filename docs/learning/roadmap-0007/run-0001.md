## Summary

Roadmap 0007 replaced the early recorder and dictionary-shaped planning
prototypes with a typed, transactional, non-executing planning system.

The final shape is:

```text
typed operator command
  -> application command service
    -> one PostgresUnitOfWork
      -> source-of-truth stores sharing one connection
        -> durable operation/session/graph/plan/approval facts

ValidatedGraph x ValidatedGraph
  -> GraphDiff
    -> ActivityPlan
      -> approval policy
        -> focused read projections
```

The package can now record a topology-editing session, publish a desired graph
with optimistic concurrency, derive and persist a typed activity plan, request
and decide approval, describe recovery honestly, and render the result through
core, FastAPI, CLI, and MCP reads. It does not execute the plan.

## Main Decisions

### One Command, One Transaction

Each application command opens one UnitOfWork. Every participating store uses
that UnitOfWork's one Postgres connection. A command requests commit only after
all source-of-truth and audit evidence is written.

```python
with self._unit_of_work_factory() as work:
    stores = work.stores
    stores.graph_topology.save(graph_record)
    stores.workspace.set_desired_graph(workspace_id, graph_record.graph_id)
    stores.activity_history.add_action(action)
    work.commit()
```

The physical commit occurs on clean context exit. Late exceptions and SQL
failures roll the complete command back. Stores cannot commit independently.

### Workflow Atomicity Is Not One Giant Transaction

Starting a session, setting desired graph truth, planning, and requesting
approval are four operator commands. They intentionally form four committed
steps. Each result becomes durable, inspectable input to the next command.

Holding one database transaction across an operator review workflow would be
incorrect: it would retain locks across human time and hide useful history.

### Closed Values Replace String Protocols

The planning model is now constrained by typed products and sums:

```text
StructuralChange
  = AddedChange
  | RemovedChange
  | ModifiedChange
  | UnsupportedChange
  | AmbiguousChange

ActivityOperation
  = StartNode
  | StopNode
  | WaitForHealthy
  | AddSocketConnection
  | SwitchSocketConnection
  | RemoveSocketConnection
  | ReconcileNode
  | ReconcileRuntime
  | StartRuntime
  | StopRuntime
  | ReviewChange
```

Pattern matching interprets these values. Versioned codecs carry them across
the JSON/Postgres boundary and reject unknown or lossy variants.

### Plans Are Historical Facts

A persisted plan records its base and desired graph IDs. Later changes to the
workspace desired pointer do not reinterpret that plan. The #160 hardening
fixture proves the plan detail and recovery projection stay pinned.

### Approval Is A Workflow, Not A Boolean

Approval request and decision are separate normalized facts with independent
idempotency and action evidence. Destructive plans derive
`plan:approve-destructive`; ordinary approval authority is insufficient.

### Recovery Is A Candidate

Recovery is a newly compiled plan toward a prior graph, accompanied by explicit
limitations. It is not an inverse operation and does not claim that data or
external effects can be restored.

## Curated Implementation Shape

The complete Roadmap 0007 example is intentionally ordinary composition:

```python
session = services.operations.execute(StartOperationSession(...))
desired = services.desired_graphs.execute(SetDesiredGraph(...))
plan = services.plans.execute(RequestActivityPlan(...))
approval = services.approvals.execute(RequestPlanApproval(...))
```

Planning itself remains pure:

```python
current = validate_graph(codec.decode(current_record.graph_descriptor))
desired = validate_graph(codec.decode(desired_record.graph_descriptor))
diff = diff_graphs(current, desired)
plan = compile_activity_plan(diff)
```

Adapters remain interpretations of one projection:

```python
api_payload = client.get(path, headers=auth).json()

self.assertEqual(api_payload, expected)
```

## Review And Hardening Findings

- Session start idempotency needed a scoped advisory lock before a session row
  existed.
- Per-session action ordinal allocation needed row locking; `len(actions) + 1`
  was not concurrency-safe.
- Desired graph updates needed optimistic pointer checks plus idempotent replay
  before treating a retry as stale.
- The graph codec had to preserve typed block specification identity rather
  than reducing future variants to metadata strings.
- Graph validation needed to return typed evidence usable by both diffing and
  persistence boundaries.
- Activity DAG construction needed cycle, missing dependency, duplicate ID,
  duplicate edge, and deterministic-order hardening.
- Planning had to recheck session state after acquiring the action-order lock.
- Approval request and decision needed distinct idempotency scopes and
  concurrency serialization.
- MCP JSON Schema could not be trusted as the only validation layer; direct
  adapter calls now enforce bounds and reject unknown arguments.
- Cross-adapter equivalence was best tested through one real-Postgres fixture,
  not four collections of mocks.
- Final prototype audit found and removed the old direct `ActivityRunService`.
  It could append a run through a store without UnitOfWork, approval, claim, or
  execution policy. The durable run record remains; Roadmap 0008 must introduce
  the real transactional execution service rather than reviving that recorder.

## Data Engineering Result

ADR 0008 is reflected in executable tests:

- all stores in one command share one connection;
- uncommitted writes are invisible to independent connections;
- late SQL failure rolls back workspace, graph, session, action, and plan facts;
- an exception after commit request still rolls back;
- idempotent retries return original durable evidence;
- conflicting retries fail explicitly;
- concurrent writers converge or serialize rather than publishing duplicates;
- and each deployed CPI is expected to inject its own database connection
  factory.

The graph descriptor remains JSON in Postgres behind `GraphTopologyStore` and
`GraphDescriptorCodec`. A future graph database adapter can replace physical
storage without changing workflow command semantics.

## Security Result

New durable and read surfaces fail closed:

- command descriptors omit operator-supplied values where appropriate;
- secret-shaped values are rejected in command evidence unless represented as
  references;
- graph, activity, event, metadata, and address-like fields are recursively
  redacted in projections;
- focused reads enforce workspace boundaries without disclosing foreign IDs;
- FastAPI reads share configurable token authentication;
- MCP remains read-only and rejects unknown/mutation-like tools;
- read bounds are enforced by both schemas and implementation;
- approval policy distinguishes request, ordinary approval, and destructive
  approval scopes;
- and Roadmap 0007 exposes no runtime execution command.

The command APIs are transport-neutral Python services. Roadmap 0009 must add
authenticated HTTP command routes without weakening these authorization and
redaction boundaries.

## Mathematical Design Note

- Objects: operation commands, graph versions, validated graphs, structural
  changes, activity plans, approval facts, recovery candidates.
- Transformations: command interpretation, descriptor decoding, validation,
  graph diffing, plan compilation, policy evaluation, projection.
- Laws: deterministic pure planning; codec round-trip fidelity; one command per
  UnitOfWork; stable plan graph references; no execution in Roadmap 0007.
- Valid compositions: typed command -> transactional service -> durable fact;
  validated graph pair -> diff -> plan; plan -> approval requirement.
- Invalid compositions: raw descriptor -> planner without validation; adapter
  -> store mutation; approval boolean without request/decision evidence;
  recovery candidate presented as guaranteed undo.

## Operational Reliability

- Health/status: not changed; runtime health belongs to Roadmap 0008.
- Logs/events: durable operation actions and plan/approval records exist;
  process logs remain supplementary.
- Failure modes: stale pointers, malformed descriptors, conflicting
  idempotency keys, authorization denial, invalid plans, and missing workspace
  truth are explicit typed errors.
- Cleanup: no runtime resources are created by this roadmap.
- Retry/resume: every command is idempotent within its documented scope.
- Example: `examples/backend_swap_planning.py` demonstrates the full planning
  workflow and explicitly reports `runtime_effects_executed = False`.
- Still missing: activity claims, runs, events around effects, runtime health,
  pause/resume, compensation execution, and live Docker mutation.

event state and saga-style compensation. It must never equate plan persistence
with execution or container start with application health.
